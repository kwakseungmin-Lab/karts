01 — System Overview

시스템 개요 & 구조

ViMax는 main 브랜치의 단편영화 전용 파이프라인과 달리, 어떤 아이디어나 스크립트든 영상으로 변환하는 범용 프레임워크입니다.
🧠

LLM 오케스트레이션

LangChain init_chat_model으로 ChatModel을 추상화합니다. GPT-4o, Gemini Flash, MiniMax M3 등 어떤 모델도 YAML 한 줄로 교체할 수 있습니다.

⚡

Python asyncio

Claude Code Workflow가 아닌 Python asyncio 기반입니다. asyncio.gather()로 카메라별 프레임 생성과 샷별 영상 생성을 동시에 실행합니다.

🔧

Config-Driven

YAML 설정 파일이 LLM·이미지 생성기·영상 생성기를 결정합니다. RenderBackend.from_config()가 class_path로 동적 인스턴스화합니다.

파일 구조

karts/ (vimax-sora 브랜치) ├── main_idea2video.py # 엔트리포인트 1: 아이디어 → 영상 ├── main_script2video.py # 엔트리포인트 2: 스크립트 → 영상 │ ├── pipelines/ │ ├── idea2video_pipeline.py # Idea2VideoPipeline (Script2Video 내포) │ ├── script2video_pipeline.py # Script2VideoPipeline (핵심 로직) │ └── novel2movie_pipeline.py # TODO — 미구현 │ ├── agents/ # LLM 기반 에이전트 클래스 │ ├── screenwriter.py # 아이디어 → 스토리/스크립트 │ ├── character_extractor.py # 스크립트 → 캐릭터 목록 │ ├── character_portraits_generator.py │ ├── storyboard_artist.py # 스크립트 → 샷 설명 목록 │ ├── camera_image_generator.py# 카메라 트리 + 프레임 생성 │ ├── reference_image_selector.py │ └── best_image_selector.py │ ├── tools/ │ ├── render_backend.py # YAML → 이미지/영상 생성기 팩토리 │ ├── image_generator_gptimage_openai_api.py │ ├── image_generator_nanobanana_google_api.py │ ├── video_generator_sora_openai_api.py │ ├── video_generator_veo_google_api.py │ ├── video_generator_omni_yunwu_api.py │ └── ... │ ├── configs/ # 프로바이더 조합 YAML 프리셋 │ ├── idea2video_sora.yaml # GPT-4o + GPT Image + Sora │ ├── idea2video.yaml # Gemini Flash + NanaBanana + Veo │ ├── idea2video_minimax.yaml # MiniMax M3 + NanaBanana + Veo │ └── idea2video_omni_yunwu.yaml │ └── interfaces/ # Pydantic 데이터 모델 ├── shot_description.py # ShotDescription, ShotBriefDescription ├── camera.py # Camera (idx, parent, active_shot_idxs) ├── character.py # CharacterInScene └── ...
ℹ️
main 브랜치와의 차이: main 브랜치는 "명예의 무게" 단편영화 한 편을 위한 고정 파이프라인이고, vimax-sora 브랜치는 어떤 내용이든 넣으면 영상이 나오는 범용 프레임워크입니다. 에이전트 정의 방식도 다릅니다 — main은 Claude Code `.claude/agents/` 마크다운, vimax-sora는 Python 클래스입니다.
02 — Entry Points

두 가지 엔트리포인트

입력 형태에 따라 두 파이프라인 중 하나를 선택합니다.
💡
Idea2Video
main_idea2video.py
idea → 스크립트 → 영상

아이디어(자연어 한 문단)를 입력하면 Screenwriter 에이전트가 먼저 스크립트를 생성하고, 이후 Script2Video 파이프라인으로 위임합니다.

# 사용 예시
idea = """
A fit woman exercises in a gym
with a beautiful beach view outside.
"""
user_requirement = "For adults, max 3 scenes"
style = "Realistic, warm feel"

pipeline = Idea2VideoPipeline.init_from_config(
    config_path="configs/idea2video_sora.yaml")
await pipeline(idea=idea,
              user_requirement=user_requirement,
              style=style)
📝
Script2Video
main_script2video.py
스크립트 → 영상

영화 스크립트 형식(INT./EXT. 씬 헤더, 대사, 액션 설명)을 직접 입력합니다. Screenwriter 단계가 없으므로 더 빠릅니다.

# 사용 예시
script = """
EXT. SCHOOL GYM - DAY
John (18, male) practices basketball.
Jane: Good job, John!
John: Yes! (ball swishes through)
"""
style = "Anime Style"

pipeline = Script2VideoPipeline.init_from_config(
    config_path="configs/script2video_sora.yaml")
await pipeline(script=script,
              user_requirement=...,
              style=style)

목표 영상 길이 자동 계산

config에 target_duration_minutes를 설정하면, 파이프라인이 필요한 샷 수를 자동 계산해 LLM 프롬프트에 주입합니다.

# 예: target_duration_minutes=7, shot_duration_seconds=8
required_shots = math.ceil(7 * 60 / 8)  # = 53샷

hint = f"The final video must be approximately 7 minutes long. "
       f"Generate enough scenes and shots to reach roughly 53 total shots."
03 — Pipeline Flow

파이프라인 전체 흐름

Script2VideoPipeline의 7단계 흐름입니다. Idea2Video는 앞에 Screenwriter 단계가 추가되고, 이후 동일한 흐름을 따릅니다.

Generation
프레임 + 영상 병렬 생성
⚡ asyncio.gather 병렬
두 종류의 태스크가 asyncio.gather()로 동시 실행됩니다. 프레임 태스크(카메라별)는 asyncio.Event로 부모→자식 순서를 보장하며, 영상 태스크(샷별)는 해당 샷의 프레임이 완성될 때까지 Event를 기다렸다가 생성합니다.
🖼️ 프레임 태스크 (카메라별)
📹 camera_0 (parent) transition video 생성
↓ Event set
📹 camera_1 (child) 부모 Event 대기
📹 camera_2 (child) 부모 Event 대기
🎬 영상 태스크 (샷별)
🎥 shot_0 video first_frame Event 대기
🎥 shot_1 video first_frame + last_frame 대기
🎥 shot_2 video first_frame Event 대기
에이전트: CameraImageGenerator 이미지 생성기 + 영상 생성기 variation_type에 따라 last_frame 생성 여부 결정

Final
편집 & 업스케일 (MoviePy + FFmpeg)
순차
MoviePy로 모든 샷 영상을 순서대로 이어붙이고, FFmpeg lanczos 필터로 1920×1080 업스케일합니다.
INPUT
working_dir/shots/{idx}/video.mp4 목록
→
OUTPUT
working_dir/final_video.mp4
1920×1080, H.264, yuv420p
MoviePy concatenate_videoclips FFmpeg upscale_inplace
04 — Camera Tree

Camera Tree — 핵심 개념

ViMax의 가장 독창적인 설계입니다. 씬에 등장하는 카메라 포지션들의 부모-자식 관계를 모델링해, 프레임 생성 시 시각적 일관성을 보장합니다.

왜 카메라 트리가 필요한가

같은 씬에 와이드샷·미디엄샷·클로즈업이 모두 있을 때, 각각 독립적으로 이미지를 생성하면 배경·캐릭터 위치·조명이 달라집니다. 카메라 트리는 부모 카메라의 시점에서 트랜지션 영상을 먼저 생성한 뒤 마지막 프레임을 추출해, 자식 카메라의 첫 프레임 생성 시 참조 이미지로 사용합니다.

트리 구성 규칙

  • 샷 크기 기준: 와이드 → 미디엄 → 클로즈업 방향으로 부모-자식 설정
  • 인접 우선: parent_shot_idx는 자식 카메라 첫 샷과 시간상 가장 가까운 샷
  • 단일 루트: 부모 없는 카메라는 반드시 1개 (트리 구조 보장)
  • 와이드→클로즈업 직접 연결 금지 (중간 미디엄샷 필수)
  • 순환 참조 금지, 인덱스 낮은 쪽이 부모

구조 예시

스크립트 씬
Shot 0: Wide shot — Alice & Bob
       walking toward each other
Shot 1: Close-up — Alice's face
       (surprise → delight)
Shot 2: Wide shot — Alice & Bob
       hug each other
카메라 트리
📹 Camera 0 (root)
shots: [0, 2] — Wide Shot
└─ 자식 카메라
📹 Camera 1 (child)
shots: [1] — Close-up
parent_shot_idx: 0

자식 카메라 첫 프레임 생성 흐름

① 부모 first_frame
parent Event 대기
→
② transition video 생성
영상 생성기: 부모 구도→자식 구도
→
③ 마지막 프레임 추출
scenedetect로 프레임 추출
→
④ 자식 first_frame 생성
추출 프레임 + 캐릭터 초상화 참조
ℹ️
missing_info 필드: 자식 카메라가 부모보다 클로즈업되어 잃는 정보(예: "Character B의 상반신")를 기록합니다. 이미지 생성 프롬프트에 "잃은 요소를 캐릭터 초상화로 교체하라"는 지시가 추가됩니다.
05 — Agent Specifications

에이전트 상세 명세

모든 에이전트는 Python 클래스입니다. LangChain ChatModel + Pydantic output parser 조합으로 구조화 출력을 강제합니다.

06 — Provider Matrix

RenderBackend 팩토리 패턴

# YAML에서 class_path로 동적 인스턴스화
def from_config(config):
    img_cfg = config["image_generator"]   # class_path + init_args
    vid_cfg = config["video_generator"]   # class_path + init_args

    # RateLimiter 생성 (RPM + RPD)
    img_rl = RateLimiter(max_requests_per_minute=img_cfg["max_requests_per_minute"], ...)
    vid_rl = RateLimiter(max_requests_per_minute=vid_cfg["max_requests_per_minute"], ...)

    # class_path → 동적 import + 인스턴스화
    module_path, cls_name = img_cfg["class_path"].rsplit(".", 1)
    cls = getattr(importlib.import_module(module_path), cls_name)
    image_gen = cls(**init_args, rate_limiter=img_rl)

    return RenderBackend(image_generator=image_gen, video_generator=video_gen)
07 — Async Architecture

비동기 실행 구조

asyncio.Event를 활용해 카메라 트리의 의존 관계를 지키면서 최대한 병렬로 실행합니다.

Event 기반 동기화

각 샷에 대해 frame_events[shot_idx]["first_frame"]과 "last_frame" Event가 생성됩니다. 프레임 생성 완료 시 event.set(), 영상 생성 태스크는 await event.wait()로 대기합니다.

# 영상 생성: 프레임 완성 대기 후 실행
await self.frame_events[shot_idx]["first_frame"].wait()
if variation_type in ["medium", "large"]:
    await self.frame_events[shot_idx]["last_frame"].wait()
video = await video_generator.generate_single_video(...)

priority_tasks vs normal_tasks

다른 카메라의 부모 샷(priority_shot_idxs)은 먼저 처리됩니다. 부모 카메라의 first_frame이 완성되어야 자식 카메라의 transition video 생성이 시작될 수 있기 때문입니다.

await asyncio.gather(*priority_tasks)  # 부모 샷 먼저
await asyncio.gather(*normal_tasks)    # 나머지 병렬

영상 생성 실패 시 Fallback

영상 생성기 호출이 실패하면 조용히 스킵하지 않고, FFmpeg still-image fallback으로 첫 프레임을 8초 정지 영상으로 변환합니다.

except Exception as e:
    print(f"⚠️ Shot {shot_idx} failed — still-image fallback.")
    subprocess.run([
        "ffmpeg", "-y", "-loop", "1", "-i", frame_paths[0],
        "-t", "8",
        "-vf", "scale=1280:720:flags=lanczos,format=yuv420p",
        "-c:v", "libx264", "-preset", "fast", "-crf", "23",
        "-movflags", "+faststart", video_path
    ], check=True)
